# Framework Inventory

Purpose:
Capture the main frameworks and sub-frameworks already present in the starter prompt and repo-native operating system.

Use this as the raw inventory, not the final lesson sequence.

## Framework families

### Orientation and reading-path frameworks

- agent-native project orientation
- always-loaded context hygiene
- change-map driven selective reading
- cold-start orientation

### Source-authority frameworks

- source corpus distillation
- source authority mapping
- derived distillation over raw archaeology

### Planning and execution frameworks

- arcs and phases
- design and implementation pairing
- lifecycle activation and closeout rules
- scratchpad and issue tracking

### Architecture and boundary frameworks

- system orientation
- schema visibility
- public vs internal boundaries
- ownership and auth seams
- file-level orientation headers

### UX and workflow frameworks

- outcome-driven UX process
- experience model
- workflow coherence review
- progressive disclosure and path-to-outcome thinking

### Validation and review frameworks

- test strategy
- smallest meaningful validation
- manual QA
- review templates
- design review gates
- security review gates
- hygiene checkpoints

### Trust and release frameworks

- security posture and dependency hygiene
- external resources and infrastructure
- release-readiness checklist
- environment and local infra hygiene

### Documentation discipline frameworks

- docs as operational memory
- canonical references, not duplicated guidance
- closeout freshness rules

## Lesson-candidate filter

A framework should become a full lesson when:

- it solves a recurring pain that readers will recognize
- it has a clear strategy, not just tactics
- the repo contains a concrete artifact that embodies it

Otherwise, it should stay as:

- a sub-lesson
- a supporting example
- or a tactic inside a larger lesson
