# Failure Family Map

Purpose:
Group the framework inventory by recurring mess patterns rather than by prompt section order.

This is the main teaching organizer.

## 1. Orientation failures

Symptoms:

- people read the wrong docs
- active state is unclear
- each new session begins with archaeology

Stabilizing frameworks:

- reading paths
- docs index
- cold-start guide
- change map

## 2. Source-truth failures

Symptoms:

- raw PRDs, chats, and notes compete for authority
- implementation starts from the wrong source
- teams re-argue old decisions

Stabilizing frameworks:

- source authority
- source distillation
- derived docs before raw corpus

## 3. Planning-state failures

Symptoms:

- active phase exists only in memory
- implementation starts before the work is bounded
- reviews and follow-up are disconnected

Stabilizing frameworks:

- arcs and phases
- lifecycle rules
- review routing
- issue register

## 4. UX coherence failures

Symptoms:

- screens grow around entities instead of user outcomes
- workflows become implementation-shaped
- features arrive before surface ownership is clear

Stabilizing frameworks:

- outcome-driven UX
- experience model
- workflow coherence review

## 5. Execution-drift failures

Symptoms:

- code changes without the right context
- docs stay stale
- validation is either too little or theatrically broad

Stabilizing frameworks:

- change map
- smallest meaningful validation
- closeout freshness rules

## 6. Trust and release failures

Symptoms:

- "works locally" gets confused with release readiness
- infra and secret assumptions stay implicit
- trust-boundary changes are handled casually

Stabilizing frameworks:

- security posture
- external resources and infrastructure
- release-readiness checklist
- security review gates
